# 新版驹马后台公共组件架构升级方案

# 现有框架模式

  • 引入模式 业务系统固定引入公共组件加载js地址-scripts-loader.js,进行公共组件的注册

# 现有架构的缺点

  • 缺乏文档说明
    组件的属性需要从源码中自己查看，提供的工具类方法也需要从源码查看
  • 二次封装不全面
    部分iview提供的属性，封装后的组件并不能使用
  • 基础环境升级困难,用不了组件的新特性
    目前组件是基于iview2.0版本封装的，此版本官方已经宣布不再维护
  • 新启项目慢
    目前没有完善的整套直接可复用框架，新建项目配置很花时间，且方式不一

# 新架构对比现有架构的优点

  • 代码优化
    按需加载，减少冗余代码
  • 便利性
    灵活升级，引用私库npm方式，组件进行版本管理，可灵活发布与升级
  • 规范化
    接口约束，引入基于promise的axios组件库，对接口的调用方式进行统一
  • 其它(建议)
    自建脚手架，GraphQL，接口查询层

# 新版公共组件对前端效率的提升可以从哪些方面体现出来

  • 所见即所得
    所使用的属性与文档一致，减少查看源码的时间
  • 接口统一管理
    减少在各个系统查找接口的时间
  • 组件更新无障碍 当业务有特殊需求可以自行发布npm包，不必等待公共组件发布

# 新版公共组件的质量问题怎么保证

  • 单元测试
    对公共方法进行单元测试，特别是工具类函数
  • 发布前测试 需要测试资源对即将发布的组件进行测试

# 旧版本能否平滑过渡到新版本？或者说各业务系统较小影响的情况下使用新版本

  • 可以平滑过度
    主要的更改在公共组件的项目中

# 旧版本切换到新版本会对各业务系统造成什么样的影响，各业务系统前端需要配合做些什么

  • 使用vue plugin的方式引入新旧组件
    业务系统需要更改一下调用组件的前缀

# 初步的里程碑计划

  • 私有库的使用
    搭建好npm私有库并启用
  • 接口统一管理
    引入axios，并用统一文件进行管理
  • 新公共组件的封装准备
    统计出需要更新的旧公共组件并制定更新计划
